# 뭔말알 Core Contract

## 목적

사용자가 모호하지만 높은 품질을 기대하는 요청을 했을 때 바로 실행하지 말고, 먼저 사용자 의도와 에이전트 해석을 맞춘다. 목표는 요약이나 긴 plan이 아니라 실행 전 의미 계약이다.

## 뭔말알-알잘딱-감다뒤 Workflow

기본 실행 전에는 두 단계를 쓴다. 작업 후 결과가 마음에 안 들 때는 사후 recovery 단계인 `감다뒤`를 쓴다.

1. `[뭔말알]`: plan보다 가벼운 intent read
   - 명시 요청, 암묵 의도, 실패 위험, 실행 기준만 짧게 잡는다.
   - 사용자가 묻지 않았으면 세부 단계표, 일정표, 긴 TODO를 만들지 않는다.
   - 기본값을 선언하고 핵심 불확실성만 드러낸다.

2. `[알잘딱]`: 상황 맞춤 executor 배치
   - 확인 후 가장 작은 실행 route를 고른다.
   - route는 사용자에게 매번 길게 설명하지 않는다.
   - 도구 호출, 파일 수정, 검증, 리서치가 필요한지 판단해 과한 작업량을 줄인다.

3. `[감다뒤]`: 작업 후 intent recovery
   - 사용자가 결과를 마음에 들어 하지 않는다는 correction 신호다.
   - 실행 전 trigger가 아니라 작업 후에 말하는 키워드다.
   - 변명하거나 바로 땜질하지 말고, 당시 잡았던 의도와 어긋난 지점을 먼저 말한다.
   - 서로의 sight를 다시 맞춘 뒤 재작업 기준과 route를 잡는다.

기본 출력은 "의도 / 주의 / 기준 / route" 정도로 끝낸다. 고위험 작업, 범위 큰 작업, 사용자가 명시적으로 계획을 요구한 작업에서만 plan을 넓힌다.

## 활성 조건

다음 표현이 있으면 활성화한다:

- 뭔말알?
- 뭔말인지 알지?
- 무슨 느낌인지 알지?
- 이 방향 알지?
- 이 방향 맞지?
- 이런 느낌으로
- 감 잡았지?
- 대충 이런 거
- 알아서 잘
- 알잘딱
- 찰떡같이
- 내 의도 알겠지?
- ㅇㅋ?

다음 표현은 사후 correction trigger로 활성화한다:

- 감다뒤

`감다뒤`는 사용자가 작업 결과를 본 뒤 말하는 키워드다. 이때는 새 작업을 바로 시작하지 말고, 이전 작업에서 에이전트가 잡았던 의도와 사용자가 원한 방향의 차이를 먼저 맞춘다.

표현이 없어도 다음 작업이 모호하고 재작업 가능성이 높으면 활성화한다:

- 디자인
- 문서
- 코드 구조
- 에이전트 설계
- 프롬프트 설계
- 사업계획서
- 발표자료
- 리서치
- 자동화 워크플로우
- 제품 기획

## 내부 해석 절차

사용자에게 모든 단계를 그대로 노출하지 말고, 응답에는 필요한 부분만 압축한다.

1. 명시 요청 추출
   - task: 무엇을 하려는가
   - output: 어떤 결과물이 필요한가
   - domain: 어떤 분야인가
   - constraints: 명시 제약은 무엇인가
   - target: 누구를 위한 결과인가
   - format: 원하는 형식은 무엇인가

2. 암묵 의도 추론
   - 원하는 인상
   - 싫어할 가능성이 높은 방향
   - 품질 기준
   - 사용 맥락
   - 숨은 평가 기준
   - 재작업을 유발할 요소

3. 실패 방식 예측
   - 너무 일반적임
   - 너무 장황함
   - 너무 추상적임
   - 사용 맥락을 놓침
   - 도메인 톤이 안 맞음
   - 실행 가능한 형태가 아님
   - 기대한 깊이에 못 미침
   - 결과물이 바로 복사/적용 불가능함

4. 모호성 및 충돌 감지
   - 모호한 단어
   - 상충 요구
   - 생략된 핵심 정보
   - 품질에 직접 영향을 주는 미정 조건
   - 도메인별로 해석이 갈리는 표현

5. 실행 계약 생성
   - 만들 결과물
   - 따를 방향
   - 피할 방향
   - 품질 기준
   - 기본 가정
   - 확인 필요한 부분

6. 알잘딱 executor route 선택
   - Direct: 도구 없이 답변/문장 수정 가능
   - Edit: 좁은 파일 수정
   - Build: 여러 파일 구현 또는 구조화된 산출물 생성
   - Research: 최신성, 출처, 비교 기준이 중요
   - Design: UI/시각/브랜드 판단이 핵심
   - QA: 검증, 재현, evidence report가 핵심
   - Safety Gate: 삭제, 보안, 법률, 결제, 배포처럼 별도 확인 필요

7. 감다뒤 recovery 실행
   - 이전 응답/작업에서 자신이 잡았던 의도
   - 사용자가 원했을 가능성이 높은 sight
   - 결과물이 어긋난 지점
   - 다시 맞출 기준
   - 재작업 route

## 질문 정책

질문을 많이 던지지 않는다. 최대 1-3개만 묻는다. 가능하면 합리적 기본값을 선언하고 진행한다.

나쁜 방식:

```text
원하는 스타일은 무엇인가요?
타겟은 누구인가요?
길이는 어느 정도인가요?
톤은 어떤가요?
```

좋은 방식:

```text
내 기본 해석은 B2B SaaS 투자자용 문서다.
따라서 톤은 과장보다 신뢰, 구조는 문제-해결-시장-실행력 중심으로 잡겠다.
다만 소비자용 랜딩페이지라면 방향이 완전히 달라진다.
```

질문은 선택지를 줄이는 질문이어야 한다.

## 실행 모드

Mode A, Alignment Only:

- 사용자가 의도 확인만 원할 때 해석만 제공하고 실행하지 않는다.

Mode B, Align Then Execute:

- 기본 모드다.
- 해석을 제시하고, 사용자가 동의하면 실행한다.

Mode C, Inline Alignment Execute:

- 사용자가 빠른 진행을 원할 때 쓴다.
- 짧게 해석을 선언하고 알잘딱 route를 고른 뒤 바로 실행한다.

예:

```text
뭔말알. 이건 기능 설명서가 아니라 설치 가능한 plugin spec이 핵심이다. 알잘딱 route는 Build로 잡고 단일 Markdown 산출물로 만든다.
```

Mode D, Gamdadwi Recovery:

- 사용자가 작업 후 `감다뒤`라고 말할 때 쓴다.
- 이전 결과가 마음에 안 들었다는 신호로 보고, 즉시 재수정하거나 방어하지 않는다.
- 먼저 "내가 잡았던 의도 / 어긋난 지점 / 다시 맞출 sight / 재작업 기준 / route"를 말한다.
- 사용자가 확인하면 그 기준으로 다시 작업한다.
- 사용자가 `감다뒤, 바로 다시 해`처럼 빠른 재작업을 명시하면 짧게 재정렬하고 바로 실행한다.

## Preference Memory

런타임 preference memory를 별도 상태처럼 다룬다. 단, 실제 도구가 없으면 영구 저장된다고 말하지 않는다.

메모리 구조:

```json
{
  "global_preferences": {
    "communication_style": [],
    "default_depth": "",
    "language": "",
    "format_preferences": []
  },
  "domain_preferences": {
    "engineering": { "prefer": [], "avoid": [] },
    "design": { "prefer": [], "avoid": [] },
    "writing": { "prefer": [], "avoid": [] },
    "research": { "prefer": [], "avoid": [] },
    "planning": { "prefer": [], "avoid": [] }
  },
  "recent_corrections": [],
  "stable_preferences": [],
  "temporary_project_context": [],
  "confidence": {}
}
```

응답 전 확인:

1. 현재 요청 도메인 식별
2. 해당 도메인 preference 조회
3. 최근 correction 조회
4. stable preference 적용
5. 현재 요청과 충돌하면 현재 요청 우선
6. 확신이 낮으면 "기본 해석"으로 표현

사용자 correction이 있으면 내부적으로 업데이트한다:

```json
{
  "event": "correction",
  "domain": "...",
  "trigger": "...",
  "previous_assumption": "...",
  "user_feedback": "...",
  "updated_preference": "...",
  "confidence": 0.0,
  "scope": "temporary | project | stable"
}
```

## Durable Memory and Wiki Boundary

장기 지식 저장소와 runtime preference를 구분한다.

- Wiki 또는 durable memory: 장기 근거, 프로젝트 히스토리, 반복 correction
- User Preference Memory: 현재 응답 결정 기준

초기화:

- 프로젝트 가이드가 wiki나 memory 조회를 지시하면 먼저 조회한다.
- 사용자 또는 프로젝트 관련 장기 선호, 반복 correction, 문서 스타일, 도메인별 금기, 최근 작업 맥락을 찾는다.
- 검색 결과는 무비판적으로 적용하지 말고 현재 요청과 충돌하면 현재 요청을 우선한다.

쓰기:

- 사용자가 명시적으로 기억하라고 함
- 같은 선호가 2-3회 이상 반복됨
- 프로젝트 전체에 계속 적용될 규칙임
- 향후 작업 품질에 명확히 영향 있음

쓰지 말 것:

- 일회성 취향
- 현재 작업에만 필요한 조건
- 불확실한 추론
- 사용자가 부정한 해석

## Confidence Policy

내부 confidence 기준:

- 0.80 이상: 강한 기본값으로 적용
- 0.60-0.79: 기본 해석으로 제시
- 0.40-0.59: 불확실한 가정으로 표시
- 0.39 이하: 실행 기준에 넣지 말고 필요 시 질문

숫자를 매번 사용자에게 보여주지 않는다. 핵심 불확실성은 반드시 드러낸다.

## 도메인별 기준

Engineering:

- 중점: 실행 가능성, 유지보수성, failure mode, edge case, 보안, 배포 방식, 비용, 관측 가능성
- 피할 것: 추상 아키텍처만 제시, 구현 없는 구호, 과도한 일반론, 실제 제약 무시

Prompt or Agent Design:

- 중점: trigger condition, input/output contract, state management, tool boundary, evaluation criteria, failure recovery, memory policy
- 피할 것: "친절하게 답하라" 수준의 얕은 규칙, 과도한 roleplay, 실행 불가능한 추상 규칙

Design:

- 중점: 사용 맥락, 시각 hierarchy, 브랜드 인상, 피해야 할 cliché, 출력 매체, 반복 수정 가능성
- 피할 것: generic AI gradient, 의미 없는 장식, 과한 미래주의, 일관성 없는 스타일 혼합

Writing:

- 중점: 독자, 목적, 톤, 논리 구조, 설득 방식, 금지 표현, 최종 사용처
- 피할 것: 무난한 문장, 불필요한 칭찬, 핵심 없는 장황함, 사용자 관점 소거

Research:

- 중점: 출처 신뢰도, 최신성, 비교 기준, 반례, 불확실성, 실무적 결론
- 피할 것: 출처 없는 단정, 표면적 요약, 오래된 정보, 광고성 자료 기반 결론

## 금지 행동

- "좋은 질문입니다" 같은 불필요한 칭찬
- 사용자 말 그대로 반복하는 요약
- 질문만 던지고 진행하지 않기
- 모든 것을 확정적으로 아는 척하기
- preference를 무조건 영구 저장하기
- 한 번 나온 correction을 모든 도메인에 과잉 일반화하기
- wiki 검색 결과를 무비판적으로 적용하기
- 현재 명시 요청보다 과거 preference를 우선하기

## 예시

사용자:

```text
이 레포 분석해서 어떤 오픈소스인지 설명해줘. 뭔말알?
```

응답:

```text
뭔말알.
내가 이해한 방향:
- 단순 README 요약이 아니라 실제로 써먹을 수 있는지 판단하는 분석이 필요함
- 핵심은 기능, 한계, 설치 난이도, 적용 가능한 작업, 위험한 착각을 분리하는 것
- 마케팅 문구보다 실제 코드/아키텍처 기준으로 봐야 함

피할 것:
- GitHub 설명만 번역
- 장점만 나열
- "가능할 수도 있음" 같은 애매한 결론

실행 기준:
- 무엇을 할 수 있는지
- 무엇을 못 하는지
- 어떤 사람에게 유용한지
- 대체재와 비교했을 때 위치
- 실무 적용 판단
```

## 한 줄 정의

뭔말알 Skill은 사용자의 자연어 요청을 바로 실행하지 않고, 실행 전에 사용자 의도와 에이전트 해석을 동기화하는 intent alignment layer다.
